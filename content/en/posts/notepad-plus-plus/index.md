This investigation started after an alert generated by an EDR: **"Possible Masquerading Behavior"**.

Possible Masquerading Behavior indicates that a process, file, or connection is impersonating another entity:

- Imitates a name  
- Imitates a path  
- Imitates behavior  
- Attempts to hide its true identity  

The detected process was:

C:\ProgramData\USOShared\svchost.exe -nostdlib -run C:\ProgramData\USOShared\conf.c

This process was launched by the following parent process:

"C:\Users\User\AppData\Roaming\Bluetooth\BluetoothService.exe" -k

Let’s analyze these processes in detail.

At first glance, the execution of `svchost.exe` from this location strongly indicates malware. We will analyze what this command does, but first we analyze the `USOShared` directory:


```c 
-drwxrwxrwx 1 root root      0 Apr  1  2024 Logs
-rwxrwxrwx 1 root root   5249 Oct 23 06:30 conf.c
-rwxrwxrwx 1 root root 148992 Oct 23 06:31 libtcc.dll
```


The executable svchost.exe is no longer present, but thanks to the EDR we were able to retrieve important information:

```c 
ImpHash: 64a1d9329f13d97b45a6c43b84ad78ca

sha256: 0f235d6df51c0f6c21d970a48fd528b8aa058ed110b1f4842391336dd324532d
```

None of these hashes have any results in VirusTotal, leaving us with little direct information. However, there are two files in this directory that are particularly interesting:

```c 
-rwxrwxrwx 1 root root 5.2K Oct 23 06:30 conf.c
-rwxrwxrwx 1 root root 146K Oct 23 06:31 libtcc.dll
drwxrwxrwx 1 root root    0 Apr  1  2024 Logs
```

Let’s start with the Logs directory:

```c 
drwxrwxrwx 1 root root   0 Apr  1  2024 .
drwxrwxrwx 1 root root   0 Oct 23 06:32 ..
drwxrwxrwx 1 root root 48K Oct 23 07:09 User
drwxrwxrwx 1 root root 64K Oct 24 17:13 System
```


By residing in C:\ProgramData\USOShared, the malware camouflages itself within the Windows Update process. USO stands for Update Session Orchestrator, a Windows Update component that normally stores update logs in this directory.

This explains the presence of the User and System directories containing .etl files. The goal of the malware here is simply to hide within legitimate Windows update processes.

We are clearly dealing with embedded shellcode, a well-known malware technique.

What does this function do?

```c
int _runmain()
{
    unsigned char temp[836] = { ... };
    ((int(*)())temp)();
}
```

- Declares a byte array
- Contains x86 machine code
- Executes the buffer directly

This is shellcode execution from RWX memory:

- x86 (32-bit)
- Windows executable
- Position Independent Code (PIC)
- Dynamic API resolving (no imports)
- Network-enabled payload

Dynamic API Resolution

```c 
FC                cld
E8 89 00 00 00    call next
60                pushad
64 8B 52 30       mov edx, fs:[edx+0x30]   ; PEB
```

The shellcode:

- Accesses the PEB (Process Environment Block)
- Walks InMemoryOrderModuleList
- Locates kernel32.dll
- Resolves function addresses by hash


A repeated hashing pattern is observed:

```c 
AC
C1 CF 0C
01 C7
```

This is a classic rotating hash used to resolve APIs such as:

- LoadLibraryA
- GetProcAddress
- WinExec
- VirtualAlloc
- InternetOpenA
- InternetConnectA
- HttpOpenRequestA
- InternetReadFile

There are no visible imports to evade static analysis. Cleartext strings related to C2 communication are present, revealing a new IoC:

```c
api[.]wiresguard[.]com
```

Full HTTP headers observed:

```c
Host: api.wiresguard.com
Accept: */*
Accept-Language: en-US,en;q=0.5
Connection: close
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) ...
``` 

This indicates a manually crafted HTTP beacon.

Remote Payload Execution

```c 
68 00 10 00 00   ; MEM_COMMIT
68 00 00 40 00   ; PAGE_EXECUTE_READWRITE
FF D5            ; VirtualAlloc
```

- Allocates RWX memory
- Copies received bytes
- Jumps to the allocated address

The string /users/admin is also observed, associated with the C2 HTTP path. High entropy suggests the payload is encrypted and decrypted at runtime.

Inside the same directory we find:

```c
4a52570eeaf9d27722377865df312e295a7a23c3b6eb991944c2ecd707cc9906  libtcc.dll
```

This library belongs to the Tiny C Compiler project. Hash comparison confirms it matches the legitimate libtcc.dll, available at:

https://download.savannah.gnu.org/releases/tinycc/

This confirms runtime code compilation.


Considering that the digital signature is valid, this file is legitimately used by Bitdefender to manage and execute subscription-related operations, including:

- Activating Bitdefender using a license key
- Renewing the license upon expiration
- Linking the endpoint to a Bitdefender Central account
- Validating the subscription status with Bitdefender servers
- Synchronizing plan and license changes

It is important to note that during the analysis, the file `log.dll` was also identified in the same directory as `BluetoothService.exe`. This strongly suggests a well-known attack technique called **DLL side-loading**, which consists of placing a malicious DLL in the same directory as a legitimate executable.

In this case, at runtime, `BluetoothService.exe` loads `log.dll` from its local directory first, resulting in the execution of malicious code from an otherwise legitimate and signed Bitdefender binary.

## Analysis of BluetoothService

The `BluetoothService` file loads the `log.dll` library:


```c
LoadLibrary("log.dll")
```

The main C2-related indicators are:

- Uses Service Control Manager (SCM) APIs
- Installs itself using a legitimate-looking name
- Avoids interactive execution

An advanced API resolution method was identified, which consists of resolving APIs using hashing techniques:


```c
ResolveAPI(hash("WS2_32.dll"), hash("connect"))
ResolveAPI(hash("kernel32"), hash("LoadLibraryA"))
```

The DLL loading method is fully manual. Instead of relying on standard loader functions, as most software would, this approach enables the malware to evade many antivirus solutions and EDR platforms.


```c 
HMODULE h = LoadLibraryA("log.dll");
fn1 = GetProcAddress(h, hash1);
fn2 = GetProcAddress(h, hash2);

```

Based on the analysis of the collected information, I can conclude that the workflow is as follows:


```c 
[ EXE ]
 ├─ Collect system info
 ├─ Pass data → log.dll
 │     └─ Encrypt / pack
 ├─ Open socket (raw / custom)
 ├─ Send blob
 ├─ Receive response
 ├─ Pass response → log.dll
 │     └─ Decrypt
 └─ Execute command
```

The `update.exe` file creates the `BluetoothService.exe` and `log.dll` files. According to the dynamic analysis performed, attempts to contact the domain `api[.]skycloudcenter[.]com` can be observed. However, at this stage, the primary point of interest is how the `update.exe` file reached the system, which leads us to identify the originating process of this execution.

We will now analyze the parent process of the `update.exe` file, and here comes the first surprise, as the parent process is:


```c
2025-10-21 14:56:29 UTC "C:\Program Files\Notepad++\updater\gup.exe" -v8.85 -px64
```

This leads us to consider a possible alteration of the `GUP.xml` file. However, no modifications were observed: the last modification date predates the incident by several months, and its hash matches the original Notepad++ file.


```c
92717951aae89e960b142cef3d273f104051896a3d527a78ca4a88c22b5216a5
```

The hash of the `GUP.exe` file also perfectly matches the legitimate version:


```c
5eb90daf1cad88ad33bceed04b0d01cb5aaf3883f991516ffc9e4b99a1c413de
```

A comprehensive analysis of all domains and network flows across firewalls and DNS logs was conducted. This analysis led to the unequivocal conclusion that `GUP.exe` performs an update check against the domain `notepad-plus-plus.com`, and that the response received resulted in the download of the malicious `update.exe` file from the following site:

http://95[.]179[.]213[.]0/update/update.exe

Once the download is completed, the same `update.exe` file is executed by the `GUP.exe` process. A DNS analysis was performed and no evidence of DNS poisoning was observed. Therefore, the conclusions regarding the origin of the malware can be reduced to two possible scenarios:

- High-level network traffic interception occurring outside the victim’s network
- A malicious update response delivered by the Notepad++ update infrastructure (given the volume of publicly reported compromises, this possibility is considered unlikely)
